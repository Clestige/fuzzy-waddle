// GAME DEVELOPMENT

Where I spend most of my time

Real-time multiplayer is a discipline of tradeoffs. Every decision is a balance between what the player sees, what the server knows is true, and what the network can actually deliver.

I work primarily on the backend side of multiplayer games: authoritative server loops, client-side prediction and reconciliation, lag compensation, interest management, and anti-cheat. The goal is always the same — make the game feel local even when it isn't, and make cheating boring by making it not work.

The web side of my work tends to be the front door for these systems — lobbies, matchmaking UIs, leaderboards, account services, the dashboards that let a team see what their servers are doing at 2am. I build those with the same care as the game logic, because a janky lobby can ruin great netcode just as easily as bad prediction can.

[01]

Authoritative Servers

The server is the only source of truth for anything that matters. Clients send inputs, the server simulates, and the result is broadcast back. Cheating becomes a question of "what can the server be tricked into believing," which is a much smaller surface.

[02]

Prediction & Reconciliation

Clients simulate ahead on their own inputs so movement feels instant, then reconcile against server snapshots when they arrive. Done well, players don't notice they're playing at 80ms. Done poorly, they rubberband and quit.

[03]

Lag Compensation

Rewind the server state to what a player saw when they pulled the trigger, then resolve the shot against that. Fair hits register. The cost is a more complex server loop and some careful edge cases — both worth it.

[04]

Interest Management

Don't tell a client about entities it can't see or interact with. Spatial hashes, grid partitioning, and visibility checks turn an O(n²) broadcast into something a single core can actually handle.

[05]

Anti-Cheat (Server-First)

Validation on everything that touches state: movement speed, action rate, inventory mutations, coordinate bounds. Client-side checks are a second layer, not the main one. The goal is making cheating not worth the effort.

[06]

Web Front-Ends

Lobbies, matchmaking, accounts, leaderboards, live ops dashboards — the web layer around a live game. Built in React/Next.js with the same latency and reliability standards as the game servers behind them.

// FEATURED CASE STUDIES

Selected work

Three systems I've built that show how I think about real-time problems. Details are anonymized where they need to be, but the engineering is real.

CASE 01

Real-Time Sync Engine

NODE.JS · WEBSOCKETS · CANVAS · INTERPOLATION
// PROBLEM

A 2D multiplayer prototype needed to feel responsive at 60–100ms ping without rubber-banding. Existing off-the-shelf netcode either felt laggy or trusted the client too much.

// APPROACH

Built an authoritative Node.js server with a fixed-tick simulation loop. Clients predict locally on their own inputs and buffer the last N inputs; the server acknowledges processed ticks and sends back canonical snapshots. Reconciliation only re-simulates from the last acknowledged tick forward, so the cost stays small. A token-bucket rate limiter guards actions, and a spatial hash drives interest management so each client only hears about nearby entities.

// OUTCOME

Smooth movement at 80ms ping with no visible rubber-banding. A single core comfortably runs a 64-player instance with headroom. The prediction layer added roughly 90 lines of client code — the rest is the server being honest about what happened.

CASE 02

Server-Authoritative Inventory & Anti-Cheat

NODE.JS · POSTGRES · REDIS · RATE LIMITING
// PROBLEM

A browser game was losing players because duplicated items were crashing the in-game economy. The original inventory logic trusted client-reported state and only validated after the fact.

// APPROACH

Rebuilt inventory as a server-authoritative system. Every item action is validated against the authoritative world state — coordinates, ownership, action rate, inventory capacity — before it commits. Pickups are bound to a specific world position and a one-time-use token. Redis-backed token buckets throttle every action type independently. A replay-grade audit log records each state transition so suspicious patterns surface in post.

// OUTCOME

Duplication exploits dropped to zero within a week of rollout. The audit log surfaced two secondary exploit vectors the team hadn't found, both patched before they went public. Player retention recovered within the month.

CASE 03

Client-Side Integrity Monitor

JS · DOM PROTECTION · SERVER VALIDATION
// PROBLEM

A competitive web game was being targeted by userscript injections that modified client state before sending network updates. The game wasn't losing to bugs — it was losing to quietly modified clients.

// APPROACH

Built a client-side integrity layer that monitors the global scope, validates expected browser APIs before each network tick, and refuses to send updates when the environment has been tampered with. Critically, the server doesn't trust the client check either — every inbound update is independently validated against what the server already knows. The client layer is a tripwire, not a lock.

// OUTCOME

Reduced the impact of common injection-based cheats to near-zero. The layered design means a bypassed client check still can't corrupt server state. A few determined attackers kept trying; the rest moved on to easier targets.

What kinds of projects do you take on? +
Mostly real-time multiplayer backends and full-stack web work, especially when the web side is the front door for a game or live service. I'm less interested in pure content sites and more in things with a backend that has to behave under real conditions.
Do you work with existing teams and codebases? +
Yes. Most of my work involves coming into an existing codebase, understanding it quickly, and either extending it or fixing what's broken. I write code that other people can maintain — clean interfaces, real comments where they matter, no clever for the sake of clever.
What's a realistic timeline for a multiplayer backend? +
A working authoritative server with prediction and reconciliation for a simple game is usually 4–8 weeks. Production-grade with anti-cheat, scaling, and observability is more like 3–6 months. Anyone telling you it's faster is selling you something that will break in production.
How do you handle anti-cheat without hurting legit players? +
Server authority first — the server never trusts the client for anything that matters. Client-side checks are a second layer, not the main one. The goal is to make cheating require enough effort that it's not worth it, without adding friction for honest players. If your anti-cheat is annoying your real users, it's designed wrong.
Do you do volunteer or open source work? +
For projects that are genuinely interesting or a chance to build something I haven't built before, yes. I'm not free labor for vague ideas, but I'm generous with my time for the right project and the right people.
